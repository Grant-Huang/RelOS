## 1. 产品背景与问题定义

### 1.1 背景

工业制造业 AI 落地失败率高达 83%（McKinsey 2025）。根本原因不是大模型能力不足，而是**架构问题**：

- 现有 AI 工具是"外挂"：读取数据、给出答案，但无法理解生产系统内部的关系结构
- 知识无法积累：工程师的确认/否认消失在操作日志里，下次遇到同样问题仍从零开始
- 缺乏可解释性：工业场景需要"为什么这么建议"，黑盒 AI 无法获得工程师信任
- 与现有系统割裂：MES/ERP 是几十年积累，AI 工具无法在其上"长出来"

### 1.2 核心问题

> **一个有 20 年经验的维修工程师知道"1 号机在高温天气下的振动报警 70% 是轴承问题，另 30% 是冷却液不足"。这个知识只在他脑子里，系统不知道，AI 不知道，新人更不知道。**

### 1.3 我们的解决方案

RelOS 不是又一个 AI 工具，而是一套**让工厂知识可计算、可推理、可演化的基础设施**：

```
Machine_M1 + HighTemp_Context --[INDICATES, confidence=0.70]--> BearingFailure
Machine_M1 + HighTemp_Context --[INDICATES, confidence=0.30]--> CoolantInsufficiency
```

这条"关系"带有置信度、来源、时间戳和上下文条件，永远不会因为工程师离职而消失。

## 4. 核心功能需求

### 4.1 关系接入与标准化（Ingestion Layer）

**FR-01：多源关系接入**
- 系统必须支持以下来源的关系接入：手动录入（工程师）、传感器实时数据、MES 结构化导入、LLM 文本抽取
- 所有来源必须收敛到统一的 `RelationObject` Schema

**FR-02：置信度初始化**
- 不同来源的关系必须按规则初始化置信度区间（详见数据模型文档）
- LLM 抽取的关系置信度硬上限 0.85

**FR-03：LLM 关系强制待审**
- 所有 LLM 抽取的关系必须进入 `pending_review` 状态，不允许直接变为 `active`

### 4.2 关系图谱管理（Relation Core Engine）

**FR-04：关系合并**
- 相同节点对、相同关系类型的新观测到来时，系统必须执行加权滑动平均合并
- alpha 权重由来源类型决定（manual_engineer=0.3，sensor_realtime=0.5，llm_extracted=0.2）

**FR-05：置信度衰减**
- 系统必须对长期未更新的关系执行指数衰减：`confidence(t) = c0 × 0.5^(t/half_life)`
- 不同关系类型有独立的半衰期配置（设备告警 90 天，物理关系 365 天）
- 衰减下限 0.05，关系不会完全消失

**FR-06：冲突管理**
- 冲突关系不删除，标注为 `conflicted` 并记录冲突 relation ID
- 置信度差异 > 0.5 的同节点对关系自动触发冲突标记

**FR-07：人工反馈**
- 工程师确认关系：confidence += 0.15，状态 → active
- 工程师否定关系：confidence -= 0.30，confidence < 0.2 则 archived

### 4.3 上下文编译（Context Engine）

**FR-08：子图提取**
- 支持以任意节点为中心，提取 N 跳子图（默认 2 跳）
- 支持置信度过滤、状态过滤

**FR-09：六层剪枝**
- 依次执行：过滤 archived → 过滤低置信度 → 优先直接关联 → 按置信度排序 → 去重 → Token 预算截断

**FR-10：Prompt 编译**
- 子图必须编译为结构化 Markdown 表格，LLM 可直接消费
- Token 预算默认 1500，可配置

### 4.4 决策推理（Decision Engine）

**FR-11：三路决策引擎**
- 高置信度（≥ 0.75）：规则引擎直接推断，零 LLM Token 消耗
- 中等置信度（0.5–0.75）：LLM 融合分析
- 低置信度（< 0.5）：触发 Human-in-the-Loop

**FR-12：LLM 分析**
- 调用 Anthropic Claude，附带关系上下文 Prompt
- LLM 输出必须是结构化 JSON（根因 + 置信度 + 推理链）
- LLM 输出的置信度上限 0.85

**FR-13：HITL 触发条件（六条规则）**
1. 子图平均置信度 < 0.5
2. 告警级别为 critical 且无高置信度历史关系
3. 冲突关系数量 > 2
4. 图中无数据（新设备）
5. 规则引擎无匹配 + LLM confidence < 0.4
6. 工程师手动触发（`force_hitl=true`）

### 4.5 执行管理（Action Engine）

**FR-14：八状态执行状态机**
- 状态：PENDING → PRE_FLIGHT_CHECK → APPROVED/REJECTED → EXECUTING → COMPLETED/FAILED → ROLLED_BACK
- 每次状态转换记录不可变审计日志（operator_id + timestamp）

**FR-15：Pre-flight Check（五步验证）**
1. 设备 ID 格式合法性
2. 操作描述非空且长度合理（5–500 字符）
3. 告警 ID 存在（非孤立操作）
4. 操作类型白名单检查（MVP：仅允许检查类操作）
5. 重复操作检查（24 小时内同设备同告警）

**FR-16：Shadow Mode**
- MVP 默认开启：Pre-flight Check 照常执行，EXECUTING 阶段只记录日志
- Shadow Mode 可通过配置 `SHADOW_MODE=false` 关闭（Sprint 3 后可用）

### 4.6 API 接口

**FR-17：REST API**
- `GET /v1/health`：健康检查
- `POST /v1/relations`：创建/合并关系
- `GET /v1/relations/{id}`：查询关系
- `POST /v1/relations/{id}/feedback`：提交人工反馈
- `POST /v1/relations/subgraph`：提取子图
- `GET /v1/relations/pending-review`：获取待审关系队列
- `POST /v1/decisions/analyze-alarm`：告警根因分析
- `POST /v1/decisions/execute-action`：触发操作执行
- `GET /v1/decisions/action/{id}`：查询操作状态
- `GET /v1/scenarios/line-efficiency`：场景7 产线效率分析
- `GET /v1/scenarios/cross-dept-analysis`：场景8 跨部门协同分析
- `GET /v1/scenarios/issue-resolution`：场景9 异常处理效率分析
- `GET /v1/scenarios/risk-radar`：场景10 企业级风险雷达
- `GET /v1/scenarios/resource-optimization`：场景11 资源配置优化
- `POST /v1/scenarios/strategic-simulation`：场景12 战略决策模拟

## 7. 用户故事

### 7.1 Epic 1：设备故障分析（MVP）

```
作为 维修工程师 张工，
当 1 号机触发振动超限告警（VIB-001）时，
我希望 系统在 10 秒内给出根因推荐（包含置信度），
以便 我能快速定位故障点，而不是凭感觉排查。

验收标准：
✓ 系统在 3 秒内返回根因推荐
✓ 推荐结果包含置信度和依据的关系 ID
✓ 我可以点击"确认"或"否定"按钮，系统立即更新图谱
✓ Shadow Mode 开启时，我能看到"建议检查轴承"但系统不会自动下发工单
```

```
作为 维修工程师 张工，
当 我发现根因推荐不准确时，
我希望 否定该推荐并补充正确根因，
以便 系统能从我的经验中学习，下次遇到同样告警时给出更准的建议。

验收标准：
✓ 否定后置信度下降 0.30
✓ 置信度 < 0.2 的关系自动归档（但保留历史）
✓ 30 天后遇到相同告警，规则引擎应给出更新后的推荐
```

### 7.2 Epic 2：专家知识录入

```
作为 高级工程师 李总，
当 我参加专家初始化会议时，
我希望 通过结构化访谈界面，将 20 年的经验录入系统，
以便 我退休后，这些知识不会随我离开而消失。

验收标准：
✓ 界面支持输入"设备-故障-部件"三元关系
✓ 支持直接设置置信度（0.0–1.0）
✓ 录入的关系立即可用于决策推理（manual_engineer 来源直接 active）
✓ 1 小时内可录入 20+ 条核心关系
```

### 7.3 Epic 3：IT 集成

```
作为 工厂 IT 管理员，
当 我需要将现有 MES 的历史告警数据导入 RelOS 时，
我希望 能通过 Excel 文件批量导入，而不需要写代码，
以便 系统快速建立初始图谱，无需等待实时数据积累。

验收标准：
✓ 支持标准 Excel 格式（列名：设备ID, 告警码, 根因, 频次, 日期）
✓ 导入完成后提供摘要报告（成功/失败/冲突数量）
✓ 不影响现有 MES 系统运行
```

## 10. 约束与假设

### 约束

- **技术约束**：MVP 阶段只使用 Excel 导入，不做 MES API 集成
- **团队约束**：核心团队 4 人（架构师、后端、AI 工程师、领域专家）
- **时间约束**：MVP 4 周，价值验证 6 个月
- **安全约束**：Shadow Mode 必须在 Sprint 3 前保持默认开启

### 假设

- 客户工厂已有基础数据（历史告警记录，可通过 Excel 提供）
- 客户有至少 1 名愿意参与专家初始化的高级工程师
- 客户 IT 基础设施支持 Docker 部署（或可接受公有云）
- Anthropic API 在客户网络环境下可访问（或使用本地模型替代）
