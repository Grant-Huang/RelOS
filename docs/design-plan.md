**版本**：v1.0

## 1. 设计目标

RelOS 当前是纯后端 API 服务，前端 UI 由 AgentNexus 或集成方构建。设计计划覆盖两个层面：

1. **API 体验设计**（当前优先级 P0）：接口设计的一致性、可预测性、错误信息友好度
2. **前端 UI 设计**（Sprint 3 启动）：工程师操作界面（HITL 审批、专家初始化、告警推荐展示）

## 5. 知识获取全生命周期设计（新增）

### 5.1 四阶段知识渠道与权重策略

系统知识获取拆分为四个可循环阶段，统一写入 `RelationObject`，但来源阶段不同、初始权重不同：

| 阶段 | 目标 | 主要来源 | 初始权重（建议） | 状态策略 |
|------|------|---------|------------------|---------|
| 阶段 1：初始化 | 快速建立公共底座 | 公开报告、论文、行业网站、公开图谱 | 0.35 | 默认 `pending_review`，需后续强化 |
| 阶段 2：调研 | 注入专家隐性经验 | 专家访谈、关系标注、关系新建 | 0.90 | 专家确认后可 `active` |
| 阶段 3：预训练 | 引入企业专属语义 | 历史文档、SOP、维修记录、日报 | 0.70 | 文档抽取默认 `pending_review` |
| 阶段 4：运行 | 持续在线学习与校准 | 日常确认/否定、人工修订、操作反馈 | 1.00（按事件衰减） | 实时更新，支持冲突标注 |

说明：
- 阶段 2 与阶段 3 允许多轮强化，不是一次性流程。
- 运行阶段可随时发起阶段 2/3 的强化任务（例如"补访谈 10 条关系"、"追加导入维修周报"）。
- 最终计算置信度时，阶段权重与来源权重、时间衰减共同生效，避免单一来源主导。

### 5.2 阶段 2（专家访谈）易用性设计重点

设计原则：**低认知负担、强引导、可快速跳过、最少必填**。

关键交互要求：
1. 访谈界面优先使用模板句（如"当 ___ 告警时，通常先检查 ___"），减少自由输入压力。
2. 每次只呈现一个微任务卡片（单条关系确认/否定/补充），控制在 15 秒内可完成。
3. 默认值可用（关系类型、置信度区间、设备上下文），专家只做修正而非从零填写。
4. 所有输入动作即时保存草稿，支持会话中断后继续。
5. 提供"我不确定"选项，避免强迫给出低质量标注。

#### 阶段 2 交互状态机（新增）

```
INIT（选择设备/范围）
  → DRAFTING（填写/自动补全）
    → VALIDATING（本地校验：必填/格式/范围）
      → SAVED_DRAFT（草稿已保存，可继续）
        → SUBMITTING（提交后端）
          → SUBMITTED（成功，进入 active 或待审队列）
          → SUBMIT_FAILED（失败：网络/校验/服务错误 → 可重试/离线队列）
```

边界态与容错：
- 网络不稳定：`SAVED_DRAFT` 本地持久化；`SUBMIT_FAILED` 进入离线队列，恢复网络自动重试。
- 专家不确定：选择“不确定”后关系进入 `pending_review`，并提示“后续运行期可再强化”。（避免强迫给结论）

### 5.3 阶段 4（运行强化）无感标注设计重点

阶段 4 的核心是让用户在业务操作中自然产生高价值标注，而不是额外增加负担。

关键交互要求：
1. 在告警处理流中嵌入二选一反馈（"推荐是否有用"），点击即形成最小标注事件。
2. 对否定操作自动弹出"替代关系建议"（可一键采纳），将纠错转化为新增知识。
3. 对高频告警自动生成批量确认建议（例如"过去 7 天同类告警关系是否批量确认"）。
4. 采用轻量激励机制（进度条、知识贡献计数），提升持续参与度。
5. 所有引导必须可关闭，避免干扰高压现场操作。

#### 阶段 4 交互状态机（新增）

```
RECOMMENDATION_SHOWN（展示推荐）
  → FEEDBACK_CLICKED（有用/无用/不确定）
    → (有用) CONFIRM_SENT → CONFIRM_ACK
    → (无用) REJECT_SENT  → REJECT_ACK
      → ALT_SUGGESTIONS_SHOWN（替代建议）
        → ALT_ACCEPTED（采纳并形成新关系/更新关系）
        → ALT_SKIPPED（跳过，不打扰）
```

批量确认（高频告警）：
- 触发条件：同一告警码 + 同设备/同产线在 7 天内重复出现 ≥ N 次（默认 N=5），且证据一致性高。
- 安全阈值：仅对 `status=active/pending_review` 且 `confidence ≥ 0.75` 的同类关系给出批量建议。
- 回滚窗口：批量确认后提供 24h “撤销”入口（撤销写入审计日志，不删除历史）。

离线与重试：
- 反馈事件写入离线队列（本地/移动端），恢复网络自动同步；同步失败可手动重试。

### 5.4 设计验收指标（新增）

| 指标 | 验收目标 |
|------|---------|
| 阶段 2 单条关系录入中位时长 | ≤ 20 秒 |
| 阶段 2 访谈完成率 | ≥ 85% |
| 阶段 4 运行中反馈触发率 | ≥ 60%（含确认/否定） |
| 阶段 4 无感标注转化率 | ≥ 30%（触发反馈后形成有效关系更新） |
| 用户主观负担评分（NASA-TLX 简化） | ≤ 35/100 |

### 5.5 角色分层验收门槛与回滚条件（新增）

为适配“老技工/老中层/高层没时间”的真实约束，验收按角色分层，且所有线上灰度必须具备回滚条件。

#### 5.5.1 一线老技工（Frontline）验收门槛

- **一屏可用（L1）**：推荐根因 + 置信度 + 下一步动作，3 秒内可读懂\n
- **最小反馈动作**：确认/否定按钮点击 1 次完成，不弹繁琐表单\n
- **操作耗时**：\n  - 查看推荐（不展开依据）≤ 10 秒\n  - 提交反馈 ≤ 3 秒（含离线队列写入）\n
- **容错**：断网时允许离线队列；恢复网络后自动同步成功率 ≥ 99%\n

回滚条件（满足任一项即回滚到 A 组 UI/流程）：\n
- 反馈提交失败率（5xx/网络同步失败）> 2%（24h 滚动窗口）\n
- 反馈触发率下降 > 15%（相对基线 A 组）\n

#### 5.5.2 老中层（Manager）验收门槛

- **解释可追溯（L2）**：能看到 2–3 条关键证据关系（`evidence_relations`），且能定位到关系 ID\n
- **阶段贡献可解释**：能看到阶段贡献（`phase_contributions`）并理解“来自专家/运行/文档”的占比\n
- **异常提示**：当 `requires_human_review=true` 时，能明确看到触发原因（不只是“需要人工”）\n

回滚条件：\n
- “解释可用性”评分 < 4.0/5（每周抽样访谈/问卷）\n
- 关键场景（告警分析）误解率 > 10%（访谈中对结论/证据理解错误）\n

#### 5.5.3 高层（Executive）验收门槛

- **一屏摘要（L1）**：`explanation_summary` 必须在 1 屏内说明“结论 + 风险/收益 + 主要证据阶段”\n
- **追责审计入口**：提供 `confidence_trace_id` 以便 IT/审计回放\n

回滚条件：\n
- 高层决策场景（S10/S11/S12）“一屏摘要”理解耗时中位数 > 30 秒\n
- 高层对结果信任度 < 3.5/5\n

## 6. 工作台双前端信息架构（Sprint 3+）

MVP 前端（`frontend/`）按 **运行时** 与 **知识训练** 双轨组织侧栏，验收对照单页原型 [`relos_workbench_v2.html`](relos_workbench_v2.html)。路由表与分组说明见 [`ux-flow.md`](ux-flow.md) **§5.1**。实现要点：

- **主题**：浅色 / 深色可切换，Design Token 以 CSS 变量 `--wb-*` 为主，避免硬编码工业深色写死全站。
- **提示标注区**：与后端语义对齐，列表过滤置信度区间，反馈统一走 `POST /v1/relations/{id}/feedback`（`engineer_id`、`confirmed`）。
- **知识层可见化**：层 1/2/3 页眉固定展示权威性条（预标注上限 0.85、专家确认可达 1.0、文档批审与 LLM 上限等），降低「权重即事实」的误解。
