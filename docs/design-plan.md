**版本**：v1.0  
**日期**：2026 年 3 月  
**负责人**：产品设计团队

---

## 1. 设计目标

RelOS 当前是纯后端 API 服务，前端 UI 由 AgentNexus 或集成方构建。设计计划覆盖两个层面：

1. **API 体验设计**（当前优先级 P0）：接口设计的一致性、可预测性、错误信息友好度
2. **前端 UI 设计**（Sprint 3 启动）：工程师操作界面（HITL 审批、专家初始化、告警推荐展示）

## 3. Sprint 3 UI 设计详细计划

### 3.1 设计系统（Week 9，3 天）

**目标**：建立工业风格设计系统，避免每个组件重复定义风格。

**交付物**：
```
Figma / design-system
  ├── 颜色规范（主色/辅助色/语义色）
  ├── 字体规范（标题/正文/标签/数字）
  ├── 间距系统（4px 基础单位）
  ├── 基础组件
  │   ├── Button（主/次/危险/禁用）
  │   ├── Badge（高可信/中等/不确定/待审核）
  │   ├── ConfidenceBar（置信度条）
  │   ├── AlarmCard（告警卡片）
  │   └── RelationItem（关系列表项）
  └── 暗色/亮色主题切换
```

**颜色规范**：

| Token | Value | 用途 |
|-------|-------|------|
| `--color-confidence-high` | `#16A34A` | 置信度 ≥ 75% |
| `--color-confidence-mid` | `#EA580C` | 置信度 50–74% |
| `--color-confidence-low` | `#DC2626` | 置信度 < 50% |
| `--color-pending` | `#2563EB` | 待审核状态 |
| `--color-archived` | `#6B7280` | 已归档 |
| `--color-surface` | `#1F2937` | 深色卡片背景 |
| `--color-bg` | `#111827` | 深色页面背景 |

### 3.2 核心组件设计（Week 9–10，6 天）

**ConfidenceBar 组件规范**：

```
输入：confidence: number (0.0–1.0)
输出：
  - 百分比文字：70%
  - 颜色：按阈值映射
  - 进度条：filled / empty 块（每 20% 一格，5 格）
  - 标签：高可信 / 中等 / 不确定

React 实现草图：
<ConfidenceBar value={0.70} showLabel={true} size="md" />
```

**AlarmRootCauseCard 组件规范**：

```
Props：
  - alarmCode: string
  - deviceName: string
  - recommendedCause: string
  - confidence: number
  - engineUsed: 'rule_engine' | 'llm' | 'hitl'
  - shadowMode: boolean
  - supportingRelations?: Relation[]
  - onConfirm: () => void
  - onReject: () => void

布局：
  ┌─ 头部：设备名 + 告警码 + 严重程度 badge ─┐
  ├─ 主体：推荐根因（大字）+ ConfidenceBar  ─┤
  ├─ 元数据：来源 + 引擎类型 + 时间          ─┤
  ├─ 可展开：依据关系列表（手风琴）          ─┤
  ├─ Shadow Mode banner（橙色提示条）        ─┤
  └─ 操作区：确认按钮 + 否定按钮             ─┘
```

### 3.3 HITL 审批界面（Week 10–11，3 天）

**设计目标**：让工程师能在 30 秒内审核并决策一条待审关系。

**页面结构**：
```
┌─ 页面标题：待审核关系 (N 条) ──────────────────┐
├─ 筛选栏：按设备 / 按关系类型 / 按置信度排序    ─┤
│                                               │
│  ┌─ 关系卡片 1 ─────────────────────────────┐ │
│  │  ALARM__INDICATES__COMPONENT_FAILURE     │ │
│  │  VIB-001 → 轴承磨损                      │ │
│  │  置信度 ██████░░░░  58%  中等             │ │
│  │  来源：LLM 从"2024年维修记录"中抽取       │ │
│  │  [查看原始文本 ▼]                         │ │
│  │  ─────────────────────────────────────── │ │
│  │  [✓ 确认] ············· [✗ 否定]         │ │

```

### 3.4 专家初始化向导（Week 11–12，4 天）

**4 步向导设计**：

```
步骤指示器：① 设备信息 → ② 录入关系 → ③ 导入历史 → ④ 完成

步骤 1：设备信息
  - 设备类型选择（下拉）
  - 设备名称（文本输入）
  - 车间/位置（文本输入）
  - [下一步 →]

步骤 2：录入关系（可多次添加）
  - 关系模板选择：
    [ 告警 → 根因 ] / [ 操作员 → 设备 ] / [ 工艺 → 质量 ]
  - 告警码（文本输入 + 自动补全）
  - 根因部件（文本输入）
  - 置信度（滑动条：0.5–1.0，步长 0.05）
  - 备注（可选，多行文本）
  - [+ 再添加一条关系]
  - [下一步 →]

步骤 3：导入历史数据（可选）
  - 拖拽上传区域（Excel，.xlsx）
  - 解析预览表格（前 10 条）
  - 字段映射确认
  - [跳过] / [确认导入]

步骤 4：完成
  - 汇总：已录入 N 条关系，导入 M 条历史数据
  - 系统状态：关系图谱已建立，Shadow Mode 开启
  - [开始验证 →]（跳转到告警模拟界面）
```

## 4. 设计评审流程

| 阶段 | 评审人 | 评审内容 | 通过标准 |
|------|--------|---------|---------|
| 线框图评审 | PM + 工程师代表 | 信息架构 + 操作流程合理性 | 工程师 5 秒内理解主操作 |
| 视觉稿评审 | PM + 前端 | 颜色规范 + 组件一致性 | 通过无障碍对比度检查 |
| 原型测试 | 真实工程师 2 人 | 任务完成率 + 时间 | 告警确认流程 < 30 秒 |
| 技术可行性 | 前端工程师 | 实现成本评估 | 无超过 3 天的单一组件 |

## 5. 设计工具与交付规范

| 工具 | 用途 |
|------|------|
| Figma | UI 设计、原型、Design System |
| FigJam | 用户旅程、流程图 |
| Storybook | 组件文档与展示 |
| React + Tailwind CSS | 前端实现 |

**设计交付规范**：
- 每个组件提供：设计稿 + 交互说明 + 边界情况（空状态、错误状态、加载状态）
- 颜色/间距使用 Design Token，不使用硬编码值
- 组件命名与代码命名一致（AlarmRootCauseCard → `<AlarmRootCauseCard />`）
