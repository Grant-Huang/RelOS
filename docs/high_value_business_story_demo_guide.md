# RelOS 高价值业务故事演示操作指南

本文档按“演示前准备 -> 每个故事逐步操作 -> 建议话术”的结构编排，目标是让演示者可以直接照着页面一步一步操作，不需要临场记忆接口或数据准备细节。

## 一、演示前准备

### 1. 启动后端依赖

```bash
docker compose up -d
```

### 2. 启动 API

```bash
uvicorn relos.main:app --reload --port 8000
```

### 3. 注入图谱演示数据

```bash
python scripts/seed_neo4j.py
python scripts/seed_demo_scenarios.py
```

### 4. 启动前端

```bash
cd frontend
npm install
npm run dev
```

### 5. 准备文档样例

用于 `企业文档标注` 页上传的样例位于：

- `data/demo/documents/cmms_workorder_m3.xlsx`
- `data/demo/documents/supplier_delay_q235.xlsx`
- `data/demo/documents/shift_handover_m3.docx`

## 二、故事 1：关键告警 5 分钟内定位首要根因

### 页面

- `告警根因分析`

### 操作步骤

1. 打开左侧导航 `分析与演示 -> 告警根因分析`。
2. 在 `演示告警 · 快速选择` 中点击 `VIB-001` 对应的 `1号机（注塑机）` 卡片。
3. 检查下方 `告警信息` 是否自动填入：
   - 告警码：`VIB-001`
   - 设备 ID：`device-M1`
4. 点击 `分析告警根因`。
5. 等待页面流式出现：
   - 推荐根因
   - 证据关系
   - 贡献度信息
   - 如有澄清问题，继续回答
6. 如果页面出现澄清问题，按照现场语境选择更接近的答案。

### 你应该讲什么

- “这里不是简单查报警码，而是把设备、告警、部件和历史经验关系一起调出来。”
- “系统先给出首要根因，再把证据关系展示给工程师，而不是只给一个黑盒答案。”
- “如果一线判断和系统不同，还可以通过后续反馈把经验继续沉淀。”

### 预期看到的结果

- 推荐根因偏向 `轴承磨损`
- 有设备触发告警、告警指向部件故障等证据关系
- 页面能完整跑完流式分析链路

## 三、故事 2：维修记录和交接班文档自动变成待审核知识

### 页面

- `知识训练 -> 企业文档标注`

### 操作步骤 A：维修工单知识沉淀

1. 打开 `企业文档标注` 页面。
2. 点击左下区域 `点击上传企业文档`。
3. 选择文件 `data/demo/documents/cmms_workorder_m3.xlsx`。
4. 等待文档进入列表，并自动选中该文档。
5. 在右侧 `标注工作区` 中查看抽取出的候选关系。
6. 点击 `全选待审` 或逐条勾选候选关系。
7. 点击 `批量通过所选` 或直接 `一键通过全部待审`。
8. 点击 `提交到图谱`。

### 操作步骤 B：供应商交期文档知识沉淀

1. 再次点击上传。
2. 选择 `data/demo/documents/supplier_delay_q235.xlsx`。
3. 等待抽取完成后，查看是否生成供应商、物料、交期延误相关候选关系。
4. 审核并提交。

### 操作步骤 C：交接班经验沉淀

1. 再次点击上传。
2. 选择 `data/demo/documents/shift_handover_m3.docx`。
3. 观察系统是否抽取出 `Machine / FailureMode / Operator` 相关候选关系。
4. 审核并提交。

### 你应该讲什么

- “一线最有价值的经验常常躺在 Excel 和 Word 里，系统的价值是把这些文档变成待审核的知识草稿。”
- “审核动作不是负担，而是一次把文档知识正式沉淀进组织知识库的过程。”
- “这比让工程师手工重新录入关系要高效得多。”

### 预期看到的结果

- 文档进入队列
- 文档状态从上传到待审核
- 候选关系可被批量通过
- 提交后文档统计变化

### 注意事项

- 当前后端真实支持上传 `xlsx/docx`，不要用 `pdf/txt`
- 如果未配置真实 LLM，系统会根据模板走内置 mock 抽取

## 四、故事 3：低置信度关系由工程师快速纠偏，形成知识飞轮

### 页面

- `运行时 -> 提示标注工作区`

### 操作步骤

1. 打开 `提示标注工作区`。
2. 如果页面为空，先确认已执行：
   - `python scripts/seed_neo4j.py`
   - `python scripts/seed_demo_scenarios.py`
3. 在顶部输入或确认 `操作员工号`，例如 `operator-night-lead`。
4. 先点击 `报警类` 筛选，查看与告警相关的待确认关系。
5. 选择 1 条低置信度关系，点击 `✓ 正确，写入`。
6. 再选择 1 条明显不合理的关系，点击 `✗ 错误`。
7. 切换到 `工单类` 或 `质量类`，继续演示分类处理。
8. 最后点击 `全部确认`，展示批量人机协同的效率。

### 你应该讲什么

- “系统不会把所有低把握结论强行写入，而是把最值得人工确认的部分提出来。”
- “这一步实际上就是把老师傅的判断动作沉淀成机器未来可复用的知识。”
- “从业务上看，这让 AI 真正成为工程师的助手，而不是替代者。”

### 预期看到的结果

- 队列中能看到 0.50–0.79 的关系
- 不同类别筛选可用
- 单条确认/否定与批量处理都可完成

## 五、故事 4：早会里快速找到瓶颈产线和跨部门卡点

### 页面

- `分析与演示 -> 产线效率与运营分析`

### 操作步骤 A：看瓶颈产线

1. 打开 `产线效率与运营分析`。
2. 观察顶部 `产线效率看板（S-07）`。
3. 找到状态为 `瓶颈` 的产线，通常是 `L2`。
4. 点击该瓶颈卡片，展开 `根因路径`。

### 操作步骤 B：看异常处理效率

1. 在同一页面向下看 `异常处理效率（S-09）`。
2. 对比不同故障类型平均处理时长。
3. 重点说明夜班和白班对比。

### 操作步骤 C：看跨部门协同

1. 在右侧查看 `跨部门协同（S-08）`。
2. 逐条讲解因果链：
   - 供应商延迟
   - 物料短缺
   - 工单阻塞
   - 总延误影响
3. 再说明 `延误责任分布`。

### 你应该讲什么

- “早会最怕的是每个部门都拿自己一块数据，最后谁都说不清全局影响。”
- “RelOS 把产线效率、异常处理和供应链协同放到同一视图中，让问题从现象直接走到因果链。”
- “这样一线主管可以更快决定今天先修哪台机器、先追哪个供应商、先处理哪个工单。”

### 预期看到的结果

- L2 被标为瓶颈产线
- 能展开根因路径
- 夜班处理效率低于白班
- 跨部门区域有清晰的因果链和延误责任分布

## 六、故事 5：发现夜班处理慢之后，系统直接给出资源动作和扩产风险

### 页面

- `分析与演示 -> 战略模拟与资源配置`

### 操作步骤 A：加载资源建议

1. 打开 `战略模拟与资源配置`。
2. 在右侧点击 `加载建议`。
3. 查看资源优化建议列表。
4. 重点查看：
   - 设备维护团队
   - 供应商管理专员
   - 夜班技能培训

### 操作步骤 B：做扩产模拟

1. 回到左侧 `扩产影响模拟（S-12）`。
2. 先将滑块停在 `+15%`，点击 `运行模拟`。
3. 说明这是“保守扩产”的风险水平。
4. 再将滑块调到 `+30%`，再次点击 `运行模拟`。
5. 对比：
   - 交付风险变化
   - 故障率变化
   - 质量风险变化
   - 综合评级

### 你应该讲什么

- “现场已经知道夜班处理慢、设备稳定性差，但最难的是决定先投哪里。”
- “这里系统不是只报风险，而是把风险映射成资源动作和 ROI 优先级。”
- “对一线负责人来说，这能支持‘先培训还是先加维保’这类真实决策。”

### 预期看到的结果

- 右侧出现资源建议及优先动作
- 左侧不同扩产比例会返回不同风险变化
- 页面会展示风险评级、影响路径和扩产前建议

## 七、推荐演示顺序

如果你要做 12–15 分钟演示，建议按下面顺序：

1. `故事 1`：用告警根因分析快速抓住一线注意力
2. `故事 2`：说明文档和经验如何变成知识资产
3. `故事 3`：展示 AI 不是黑盒，而是能被工程师纠偏
4. `故事 4`：把单点问题放大到产线和跨部门协同
5. `故事 5`：收尾到资源动作和扩产风险，体现业务价值闭环

## 八、演示失败时的排查顺序

1. 看后端健康检查：`/v1/health`
2. 确认已执行两个 seed 脚本
3. 确认 Neo4j 里已有基础节点与场景节点
4. 文档页如果抽取失败，先确认上传文件格式是 `xlsx/docx`
5. 若关闭了 `ALLOW_LLM_MOCK`，确认已配置 `ANTHROPIC_API_KEY`
